# Supermetrics Data Activation - Server Tag

This Google Tag Manager **Server-Side** template is the execution engine of the Supermetrics Data Activation stack. It runs inside your **Server-Side GTM container** and handles two distinct responsibilities:
1. writing a secure, long-lived first-party cookie to the user's browser
2. forwarding structured event data to the Supermetrics Data Activation tracking endpoint.

## Prerequisites

Before configuring this tag, you will need:

- A working **Server-Side GTM container** with the [**Supermetrics Data Activation - Server Client** 🔗](https://github.com/orgs/Supermetrics-Professional-Services/repositories) installed and configured.
- Your **Site ID**: a unique number identifying your CDP instance.

---

## Choosing Your Tag Type Configuration

This template handles four completely distinct execution paths. You select the mode using the **"Choose Tag Type Configuration"** dropdown at the top of the tag. You will typically create **two separate tags** using this template in your server container: one for cookie management, and one for event forwarding.

---

### Mode 1: Create Server-Set Cookie

**What it does:** Writes a secure, HTTP-only, first-party `_svtri` cookie directly to the user's browser through your server domain. This is the primary defence against browser tracking restrictions like Safari's Intelligent Tracking Prevention (ITP), which deletes JavaScript-set cookies after just 7 days. A server-set cookie issued from your own domain can persist for up to 400 days.

**When to use:** Create exactly **one** dedicated tag using this mode in your server container.

**Configuration fields:**

| Field | Description | Example |
|---|---|---|
| Cookie Domain | Your root domain with a leading dot, so the cookie is accessible across all subdomains. | `.yourdomain.com` |
| Cookie Max Age | The lifespan of the cookie in seconds. The default of `34560000` (~400 days) is strongly recommended. | `34560000` |

**Recommended trigger:** Set this tag to fire on a **Custom Event** named exactly `smda_identity`. The Server Client fires this event specifically when it processes an identity ping from the Web Tag, so this tag will execute immediately and correctly after identity resolution.

**Important:** This mode is completely self-contained. When it fires, it reads the resolved UUID and cookie name directly from the `smda_identity` event data, sets the cookie, and exits. It performs no API calls and requires no credentials.

---

### Mode 2: Inherit from Data Activation Web Tag

**What it does:** The zero-configuration event-forwarding mode. This tag automatically reads all the structured data (`x_smda_type`, `x_smda_properties`, `event_name`) that the Server Client has already parsed and packaged. It dynamically determines whether the incoming event is an Engagement, a Fact, or a Mapping, constructs the correct tracking URL, and sends it to the CDP endpoint. No manual field mapping is required.

**When to use:** This is the standard mode for routing events that originated from the Web Tag. Create one tag using this mode to handle all your web-originated events.

**Configuration fields:**

> No fields are required. The tag reads `x_smda_uuid` and `x_smda_site_id` directly from the event data generated by the Server Client.

**Prerequisites:**
- The Data Activation Web Tag must be installed in your Web GTM container.
- The **"Send events to Server-Side GTM"** option must be enabled for each event type in the Web Tag settings.

**Recommended trigger:** Use a **Custom Event** with a Regex match set to `.*` to catch all event types passed by the Server Client (e.g., `page_view`, `purchase`, `fact_name`, `mapping`). Alternatively, create specific triggers for each event name you send.

---

### Mode 3: Manual Configuration

**What it does:** Mirrors the configuration UI of the Web Tag, allowing you to bypass the Server Client's inherited data entirely and define event payloads manually. This is useful if you want to leverage existing incoming requests and events (i.e. from an existing GA4 implementation), or for sending server-side-only events that have no corresponding Web Tag event, such as CRM-triggered engagements or backend fact updates.

**When to use:** When you need to send events to the CDP directly from the server container, sourced from GA4 events or other server-side data, without going through the Web Tag pipeline.

**Configuration fields (always required):**

| Field | Description | Example |
|---|---|---|
| Data Activation Site ID | Your unique CDP site identifier. | `1234` |
| User Tracking Identifier - UUID | The user's UUID. Read it from the server-set cookie using a GTM 1st Party Cookie variable. | `{{cookie - _svtri}}` |

**Sub-type selection:** Within this mode, a second dropdown lets you choose the event type:

#### • Engagement
Tracks a named user interaction event at the CDP.

| Field | Description | Example |
|---|---|---|
| Engagement Name | the engagement event name you want to track. | `page_view` |
| Engagement Properties | A name-value list of relevant properties to send with the engagement. | Property Name: `page_url`, Property Value: `{{Page URL}}` |

#### • Fact
Stores a long-term user attribute with an automatic expiry.

| Field | Description | Example |
|---|---|---|
| Fact Name | The name of the fact as configured in your Data Activation instance. | `crm_loyalty_tier` |
| Fact Time To Live | The expiry time in seconds. After this, the CDP automatically removes the fact. | `63072000` (2 years) |
| Fact Properties | A name-value list of relevant properties to store as a fact. | Property Name: `tier_level`, Property Value: `gold` |

#### • Mapping
Associates an additional identifier (e.g., a hashed email) with the user's anonymous UUID profile.

| Field | Description | Example |
|---|---|---|
| Partner Type Slot ID | The numeric slot ID configured in Data Activation for this identifier type. | `7001` (email hash) |
| Partner Identifier | The actual identifier value to store on the slot above. | `{{SHA-256 Email Hash}}` |
| Merge Profiles | If checked, the CDP merges any existing profile associated with this ID into the user's current profile. Only enable for primary identifiers (e.g., email upon login). | — |

### Mode 4: Map from standard GA4 Ecommerce

**What it does:** Listens to standard GA4 e-commerce server-side events and automatically translates the `items` array into the specific Engagement formats expected by the Data Activation CDP. For `purchase`, `view_cart`, and `begin_checkout` events, it generates both a summary and individual item detail requests. All events fire as Engagements.

**When to use:** When you have a standard GA4 web setup (using the GA4 config tag and an `items` array in your dataLayer) and want to route e-commerce data to the CDP without using the Web Tag pipeline.

> Because this is a Server-Side tag, it reads **Event Data** generated by your incoming GA4 requests (processed by the GA4 Server Client in your ssGTM container). It does **not** read from the browser dataLayer. Ensure your web setup follows [Google Analytics recommended e-commerce standards](https://developers.google.com/analytics/devguides/collection/ga4/reference/events?client_type=gtm#online_sales), otherwise the required product parameters (like `items` and `transaction_id`) will be missing from the server-side event data.

**Supported GA4 events:** `view_item`, `add_to_wishlist`, `add_to_cart`, `remove_from_cart`, `view_cart`, `begin_checkout`, `purchase`.

> The tag includes a built-in whitelist validation. If the incoming GA4 event is not on the supported list, or if the `items` array is missing or empty, the tag exits gracefully without sending any request.

**Configuration fields:**

| Field | Description | Example |
|---|---|---|
| Data Activation Site ID | Your unique CDP site identifier. | `1234` |
| User Tracking Identifier - UUID | The user's UUID. Read it from the server-set cookie using a GTM 1st Party Cookie variable. | `{{cookie - _svtri}}` |

**`purchase`, `view_cart`, and `begin_checkout` event mapping:**

For these events, the tag sends two types of Engagements:

- **`{event}_summary`** (one per event, e.g. `purchase_summary`, `view_cart_summary`, `begin_checkout_summary`): includes `o_item_count`, `o_categories`, `o_avg_price`, and — when available in the event data — `o_id`, `o_total`, `o_currency`, `o_coupon`, `o_is_first`, `o_tax`, `o_shipping`. Properties only present in purchase events (such as `o_id`, `o_tax`, `o_shipping`) are automatically excluded for `view_cart` and `begin_checkout`.
- **`{event}_item_details`** (one per line item, e.g. `purchase_item_details`, `view_cart_item_details`): includes `i_id`, `i_name`, `i_category`, `i_price`, `i_quantity`, `i_brand`, `i_variant`, `o_item_count`, `i_line_total`, and — when available — `o_id`, `o_total`, `o_currency`, `i_coupon`.

For all other supported events (`view_item`, `add_to_wishlist`, `add_to_cart`, `remove_from_cart`), one Engagement per item is sent using the GA4 event name as the Engagement name (e.g., `add_to_cart`), with the standard item fields only.

## Tracking Endpoint Reference

The tag sends GET requests to the Supermetrics Data Activation tracking endpoint. No authentication is required. The requests are formatted as URL parameters:

| Event Type | URL Pattern |
|---|---|
| Engagement | `https://t.svtrd.com/t-{siteId}?i={uuid}&e=true&et={name}&cup={key}%3A{value}...` |
| Fact | `https://t.svtrd.com/t-{siteId}?i={uuid}&f=true&ft={name}&fttl={ttl}&cup={key}%3A{value}...` |
| Mapping | `https://t.svtrd.com/syncResponse?ca_site={siteId}&ca_partner={type}&ca_cookie={uuid}&ca_read=pid&pid={id}&ca_merge={flag}` |

> Custom properties are appended as `&cup={key}%3A{value}` parameters. Properties with null, undefined, or empty values are automatically excluded.

## Recommended Server Container Setup

For a full server-side Data Activation deployment, you should have **two tags** using this template:

| Tag Name | Mode | Trigger |
|---|---|---|
| SMDA - Set UUID Cookie | Create Server-Set Cookie | Custom Event: `smda_identity` |
| SMDA - Forward Web Tag Events | Inherit from Data Activation Web Tag | Custom Event: `.*` (Regex) or specific event names |

---

## Security Notes

This template uses standard GTM Sandboxed Server-Side JavaScript APIs only:

- `setCookie` is used exclusively to write the `_svtri` first-party identifier.
- `sendHttpRequest` is strictly scoped to the `t.svtrd.com` tracking domain.
- No authentication credentials are required or transmitted.
- Logging is enabled in debug mode only.
